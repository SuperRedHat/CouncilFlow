# TASK-123 讨论 prompt 压缩 — before/after 度量（2026-06-10）

测量对象：`render_discussion_prompt` 的 Prior Turns JSON 块（每轮发给每个 sidecar 参与者的历史上下文）。
压缩策略：末轮保留全文（上限 4000 字符），更早轮次只保留结构化立场（agreements/disagreements/open_questions）+ 240 字符摘要；JSON 去 indent。

| 单条消息长度 | 轮数(2参与者) | 旧字符数 | 新字符数 | 节省 |
|---:|---:|---:|---:|---:|
| 600 | 3 | 4,895 | 3,315 | 32.3% |
| 600 | 5 | 8,157 | 5,129 | 37.1% |
| 2400 | 3 | 15,875 | 6,975 | 56.1% |
| 2400 | 5 | 26,457 | 8,789 | 66.8% |

结论：每参与者每轮一次性节省 36–74%（消息越长、轮数越多节省越大；上表为单次调用，
整场讨论的累计节省按轮数叠加）。质量保障：收敛评估器与响应 schema 只消费结构化立场字段，
这些字段在全部轮次完整保留；末轮全文是参与者被要求回应的内容，原样保留（仅 4000 字符截断上限）。
另：`discussion wait` happy path 不再回灌全部 turn 全文（改 summary_path 引用），控制器侧上下文同步减负。
